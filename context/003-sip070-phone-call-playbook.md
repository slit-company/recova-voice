# 003 — SIP/070 공급자 전화 문의 플레이북

- 작성일: 2026-07-09
- 상태: active
- 연결 문서:
  - [`001-own-telephony-infra.md`](./001-own-telephony-infra.md)
  - [`002-korean-sip070-contract-guide.md`](./002-korean-sip070-contract-guide.md)
- 목적: 기술을 잘 모르는 사람도 공급자에게 전화해서 Recova에 맞는 070/SIP 회선을 찾고, 상사에게 쉽게 보고할 수 있게 한다.

## 배경

Recova는 고객별로 070 번호를 배정하고, 그 번호로 들어온 전화를 Recova AI 상담 워크플로우로 연결하려고 한다. 고객에게 통신사 계정이나 SIP 비밀번호를 직접 주는 구조가 아니라, **Recova가 전화 인프라를 운영하고 고객은 배정된 번호를 워크플로우에 연결만 하는 구조**다.

그래서 우리가 찾는 것은 단순한 “인터넷전화기 1대”가 아니다.

우리가 찾는 것은:

```text
070 번호
  → 공급자 SIP trunk / DID 라우팅
  → Recova 서버/Jambonz/SBC
  → Recova AI 전화 워크플로우
```

쉬운 말로 하면:

> “전화번호만 사려는 게 아니라, 그 전화번호로 걸려온 전화를 우리 서버로 보내주고, 우리 서버가 그 번호로 전화를 걸 수 있게 해주는 업체”를 찾는 것이다.

## 조사 근거

- `002-korean-sip070-contract-guide.md`에서 2트랙 전략을 확정했다.
  - 빠른 검증: 중소형 070/SIP/PBX 업체 우선.
  - production 후보: SKB/KT/LG U+ 기업영업 병행.
- 사용자 KT 통화 기록에서 확인된 점:
  - “070 번호 하나 개발용으로 받고 싶다”라고 말하면 일반 기업 인터넷전화/Centrex로 라우팅된다.
  - 일반 Centrex/전화기 설치형은 Recova가 원하는 server-to-server SIP trunk가 아니다.
  - KT 쪽에서도 “그건 PBX/교환기 영역”이라는 취지로 답했다.
- 후보 상세 리스트:
  - JS Solution / 온누리070 / webs.co.kr
  - 와이즈070 / 삼성 Wyz070 IP-LINK
  - KT 기업가입센터
  - DSN / Samsung PBX SI
  - 블루베이네트웍스
  - 엠투넷
  - iPECS 파트너
  - SKB/LG U+ 기업영업

## 결정

### 전화 문의 목표

전화의 목표는 “가입”이 아니다. 첫 통화 목표는 아래 3개를 확인하는 것이다.

1. **우리 서버로 inbound routing이 가능한가?**
   - 070 번호로 전화가 오면 Recova 서버/Jambonz/SBC로 SIP INVITE를 보내줄 수 있는가?

2. **우리 서버에서 outbound 발신이 가능한가?**
   - Recova 서버가 SIP trunk를 통해 전화를 걸 수 있는가?

3. **그때 발신번호가 우리가 받은 070 번호로 표시 가능한가?**
   - 아무 번호가 아니라, 공급자가 Recova에 준 번호를 caller ID로 쓸 수 있는가?

이 3개 중 하나라도 안 되면 Recova V1에는 맞지 않는다.

### 전화 문의 순서

1. JS Solution / 온누리070
2. 와이즈070 IP-LINK
3. KT 기업가입센터
4. DSN 또는 iPECS/Samsung PBX SI
5. SKB/LG U+ 기업영업

빠른 실통화 검증은 1~2번에서 나올 가능성이 높고, production/main trunk는 3~5번에서 나온다.

## 상사에게 쉽게 설명하는 말

### 30초 설명

```text
우리가 필요한 건 070 전화기 가입이 아니라, 070 번호를 우리 AI 전화 서버에 직접 연결해주는 회선입니다.

일반 인터넷전화는 전화기를 꽂아서 사람이 받는 상품이고,
우리가 필요한 건 전화가 오면 Recova 서버로 들어오고,
Recova 서버가 그 번호로 전화를 걸 수 있는 SIP trunk/BYOC 구조입니다.

그래서 작은 070/SIP/PBX 업체로 빠르게 실제 통화 검증을 하고,
동시에 SKB/KT/LG 같은 대형 통신사는 production 후보로 비교하려고 합니다.
```

### 더 쉬운 비유

```text
일반 070 전화 가입은 “전화기 한 대를 사는 것”입니다.
우리가 필요한 건 “전화국 선을 우리 콜센터 서버에 꽂는 것”입니다.

전화기 상품을 사면 Recova AI가 전화를 직접 받을 수 없습니다.
SIP trunk/DID 라우팅을 받아야 Recova 서버가 전화국처럼 전화를 받고 걸 수 있습니다.
```

### 상사가 물을 만한 질문과 답

| 질문 | 쉬운 답 |
|---|---|
| 왜 KT 070 하나 가입하면 안 돼? | 그건 전화기용 상품이라 우리 서버로 직접 연결이 안 될 수 있다. 우리는 AI 서버로 전화가 들어와야 한다. |
| 왜 작은 업체부터 보나? | 작은 SIP/PBX 업체가 Asterisk, SIP 계정, IP-PBX 연동을 더 빨리 이해할 가능성이 높다. 첫 목표는 production 계약이 아니라 실제 전화선 검증이다. |
| 대기업 통신사는 안 보나? | 본다. 다만 대기업은 상담 라우팅이 느리고 초기 최소 회선 수가 클 수 있다. 장기 production 후보로 병행한다. |
| ClawOps는 왜 안 쓰나? | 이번 V1 방향은 ClawOps 의존을 제거하고 Recova 자체 전화 인프라를 만드는 것이다. ClawOps는 비교 기준으로만 본다. |
| 성공 기준은 뭐야? | 070 번호로 Recova 서버에 전화가 들어오고, Recova 서버가 같은 번호로 발신하고, CDR/실패 로그가 남으면 1차 성공이다. |

## 전화 전 준비물

전화하기 전에 아래를 메모장에 열어둔다.

```text
회사명: 슬릿컴퍼니 / Recova
사업자번호: 882-81-03956
용도: 한국 B2B AI 전화 서비스 개발 및 운영
필요한 것: 070 DID + inbound SIP routing + outbound SIP trunk
초기 수량: 테스트 번호 1~5개
초기 동시콜: 1~3콜이어도 됨. 가능하면 10콜 견적도 확인
서버 위치: 클라우드 서버, GCP Seoul 예정
연동 대상: Jambonz/SBC 또는 Asterisk/FreePBX 호환 SIP endpoint
중요 조건: 고객 단말용 전화기 설치가 아니라 server-to-server SIP 연동
```

아직 모르는 값은 “아직 미정이고, 연동 가능 여부를 먼저 확인하는 단계”라고 말한다.

## 전화 첫 문장

상담원이 받으면 이렇게 시작한다.

```text
안녕하세요. 기업용 070 번호와 SIP trunk 연동 가능 여부 문의드립니다.

저희는 070 전화기 한 대를 설치하려는 게 아니라,
자체 cloud PBX/Jambonz/SBC 서버에 070 번호를 연결하려고 합니다.

070 번호로 전화가 오면 저희 서버로 SIP inbound routing이 되고,
저희 서버에서 같은 070 번호로 outbound 발신도 가능한 상품이 있는지 확인하고 싶습니다.

Asterisk, FreePBX, Jambonz 같은 외부 IP-PBX 연동 가능 여부를 아시는 기술 상담이나 기업 SIP 담당자분께 연결 가능할까요?
```

## 절대 이렇게 말하지 말 것

아래처럼 말하면 상담이 엉뚱한 곳으로 간다.

| 말하지 말 것 | 왜 안 좋은가 |
|---|---|
| “070 번호 하나 받고 싶어요” | 일반 인터넷전화기 가입으로 안내된다. |
| “개발용 번호 하나만 필요해요” | 전화기/앱용 070으로 이해할 수 있다. |
| “서버 주소랑 아이디 비밀번호 받을 수 있나요?” | 상담원이 일반 전화기 IP 주소로 오해할 수 있다. |
| “인터넷전화 가입 문의입니다” | Centrex/전화기 설치 상품으로 빠진다. |
| “콜봇 만들려고요”만 말하기 | AI 콜봇 플랫폼 영업으로 빠지고 SIP trunk 조건 확인이 흐려질 수 있다. |

## 핵심 질문 10개

전화 중 아래 질문을 순서대로 물어본다. 다 물어보지 못해도 1~5번은 반드시 확인한다.

### 1. 외부 IP-PBX 연동 가능 여부

```text
Asterisk, FreePBX, Jambonz 같은 외부 IP-PBX/SBC에 직접 SIP trunk로 연동할 수 있나요?
```

좋은 답:

```text
가능합니다. SIP 계정 방식 또는 고정 IP 인증 방식으로 연동합니다.
```

나쁜 답:

```text
전화기를 설치해야 합니다.
앱이나 단말기에서만 쓸 수 있습니다.
저희 장비 안에서만 됩니다.
```

### 2. Inbound DID 라우팅

```text
070 번호로 전화가 들어오면 저희 서버의 SIP endpoint로 바로 보내줄 수 있나요?
```

좋은 답:

```text
가능합니다. 고객 SIP URI 또는 고정 IP로 INVITE를 보낼 수 있습니다.
```

나쁜 답:

```text
전화기나 교환기 장비로만 착신됩니다.
착신전환만 가능합니다.
```

### 3. Outbound SIP 발신

```text
저희 서버에서 SIP trunk를 통해 010/02/031 같은 국내 번호로 발신할 수 있나요?
```

좋은 답:

```text
가능합니다. 국내 유선/휴대폰 발신 가능합니다.
```

나쁜 답:

```text
수신만 됩니다.
앱에서만 발신됩니다.
```

### 4. Caller ID

```text
발신할 때 공급자에서 받은 070 번호를 발신번호로 표시할 수 있나요?
```

좋은 답:

```text
가능합니다. 등록된 번호만 발신번호로 쓸 수 있습니다.
```

나쁜 답:

```text
발신번호는 임의 지정입니다.
발신번호 표시 보장 안 됩니다.
다른 번호가 나갈 수 있습니다.
```

### 5. 인증 방식

```text
연동은 SIP REGISTER 방식인가요, 아니면 고정 IP 인증 방식인가요?
```

둘 다 가능하다. Recova는 가능하면 **고정 IP 인증(IP-auth)** 이 좋다.

쉬운 설명:

- REGISTER: 아이디/비밀번호로 로그인하는 방식.
- IP-auth: 우리 서버 IP를 등록해두고 그 IP에서 오는 통화만 허용하는 방식.

### 6. 테스트 가능 여부

```text
처음에는 테스트 번호 1~5개, 동시통화 1~3개 정도로 PoC가 가능한가요?
```

좋은 답:

```text
가능합니다. 테스트 계정이나 소량 개통 가능합니다.
```

나쁜 답:

```text
최소 30채널/50회선부터 가능합니다.
```

단, production 후보로는 최소 회선이 커도 완전 탈락은 아니다.

### 7. Codec / DTMF

```text
지원 codec과 DTMF 방식은 어떻게 되나요? G.711, RFC2833 또는 SIP INFO를 지원하나요?
```

기술을 몰라도 그대로 읽으면 된다.

상담원이 되물으면:

```text
저희 기술팀이 확인할 항목이라서, 연동 문서나 스펙으로 받을 수 있는지 확인 부탁드립니다.
```

### 8. CDR / 실패 로그

```text
통화 후 CDR이나 실패 원인 로그를 받을 수 있나요?
```

좋은 답:

```text
통화내역, duration, 실패 사유, 과금 정보를 제공합니다.
```

나쁜 답:

```text
통화 성공/실패 상세는 알 수 없습니다.
```

### 9. 장애 대응

```text
장애가 나면 기술 담당자나 NOC 연락 채널이 있나요?
```

좋은 답:

```text
장애 접수 채널, 담당자, 처리 SLA가 있습니다.
```

### 10. 문서 제공

```text
가능하다면 SIP 연동 문서, 샘플 설정, 요금표, 계약 조건을 메일로 받을 수 있을까요?
```

전화의 최종 목표는 이 문서/메일을 받는 것이다.

## 상담원이 엉뚱하게 안내할 때 되돌리는 말

### “전화기 설치해야 번호가 나옵니다”라고 하면

```text
전화기 설치형 상품은 저희 목적과 조금 다릅니다.
저희는 물리 전화기가 아니라 자체 IP-PBX/SBC 서버와 SIP로 연동하려고 합니다.
외부 PBX 연동형, SIP trunk, DID/DOD, IP-LINK 같은 상품 담당자와 통화할 수 있을까요?
```

### “인터넷도 같이 가입해야 합니다”라고 하면

```text
인터넷 회선 결합 상품이 아니라, 클라우드 서버와 SIP로 연동하는 구성을 확인하고 있습니다.
고정 공인 IP 기반 연동이나 SIP REGISTER 방식이 가능한지 확인하고 싶습니다.
```

### “그건 교환기 업체에 물어보세요”라고 하면

```text
네, 교환기/SBC는 저희가 준비할 수 있습니다.
다만 귀사에서 070 DID와 outbound trunk를 외부 PBX로 넘겨줄 수 있는지가 필요합니다.
가능하다면 귀사와 연동 경험이 있는 PBX/SI 파트너를 소개받을 수 있을까요?
```

### “무슨 말인지 모르겠다”라고 하면

```text
기업용 IP-PBX, SIP trunk, DID/DOD, 콜센터 회선, BYOC 연동 담당자분께 연결 부탁드립니다.
일반 인터넷전화기 가입 문의는 아닙니다.
```

## 통화 중 판정표

통화가 끝나기 전에 아래처럼 판단한다.

| 판정 | 조건 | 다음 행동 |
|---|---|---|
| Green | inbound SIP 가능 + outbound SIP 가능 + 070 caller ID 가능 + 테스트 소량 가능 | 견적서/스펙 요청 후 기술팀 연결 |
| Yellow | SIP 가능하지만 최소 회선 큼, 또는 문서 부족 | production 후보로 보류, 다른 업체도 계속 문의 |
| Red | 전화기/Centrex만 가능, 외부 PBX 불가, CDR 없음, 발신번호 불명확 | V1 후보에서 제외 |
| Unknown | 상담원이 이해 못 함 | 기술 담당자 재연결 요청 또는 메일 문의로 전환 |

## 기술 용어 쉬운 설명

| 용어 | 쉬운 설명 |
|---|---|
| 070 DID | 고객이 거는 실제 전화번호. 이 번호로 전화가 들어온다. |
| SIP | 인터넷으로 전화 신호를 주고받는 표준 방식. |
| SIP trunk | 통신사 전화망과 우리 서버를 연결하는 전화선 같은 것. |
| IP-PBX | 인터넷 기반 사내 교환기. Recova는 이 역할을 Jambonz/SBC/서버가 한다. |
| Jambonz | Recova가 전화 통화를 AI 워크플로우에 연결하기 위해 쓰는 전화 서버 계층. |
| SBC | 통신사와 우리 서버 사이에서 SIP/RTP를 안전하게 중계하는 장비/서버. |
| inbound | 고객이 Recova 번호로 전화 거는 것. |
| outbound | Recova가 고객에게 전화 거는 것. |
| caller ID | 전화를 걸 때 상대방에게 보이는 발신번호. |
| CDR | 통화 기록. 통화 시간, 성공/실패, 비용, 종료 사유 같은 데이터. |
| codec | 음성을 압축해서 보내는 방식. 통화 품질에 영향이 있다. |
| DTMF | 전화 키패드 입력 신호. ARS에서 “1번 누르세요” 같은 기능에 필요하다. |
| REGISTER | 아이디/비밀번호로 SIP 서버에 로그인하는 방식. |
| IP-auth | 등록된 서버 IP만 통화할 수 있게 허용하는 방식. 운영 서버에는 이 방식이 더 깔끔하다. |
